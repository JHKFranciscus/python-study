# 2026-07-24 Day19

## 오늘 한 일

- 파이썬 모듈과 import의 기본 구조를 학습했다.
- JSON 도서 관리 프로그램을 main, service, storage 파일로 분리했다.
- main에서 사용자 입력과 출력, service 호출과 저장 시점 결정을 담당하게 했다.
- service에서 입력값 정리, 유효성 검사, 도서 추가·검색·수정·삭제를 담당하게 했다.
- storage에서 JSON 파일 불러오기와 저장을 담당하게 했다.
- 추가·수정·삭제가 성공했을 때만 JSON 파일을 저장하게 수정했다.
- 각 파일에서 맡지 않아야 하는 기능이 들어 있지 않은지 검색으로 확인했다.
- JSON 전체 저장 방식의 시간 복잡도를 정리했다.
- 기존 도서 관리 코드를 보지 않고 모듈형 메모 관리 프로그램을 독립 재현했다.
- 빈 제목, 문자 가격, 음수 가격, 존재하지 않는 도서 수정과 삭제를 시험했다.

## 막힌 부분

- 함수와 파일을 분리할 때 input, 데이터 처리, 파일 저장을 어느 파일에 두어야 하는지 헷갈렸다.
- main에서 함수의 반환값을 받기 전에 검색 결과 변수를 검사하려 했다.
- load_memos에서 json.load(file)의 결과를 반환하지 않았다.
- save_memos에서 파일 이름 대신 memos 리스트를 open에 전달했다.
- 메모를 검색할 때 memo 딕셔너리의 text 값이 아니라 딕셔너리 자체를 검사했다.
- lower 뒤에 괄호를 붙이지 않아 문자열이 아니라 메서드 자체가 저장되었다.

## 시도한 것

- main, service, storage의 역할을 먼저 나눈 뒤 기능을 하나씩 연결했다.
- 추가·수정·삭제 함수에서 성공 여부를 True와 False로 반환했다.
- main에서 반환값을 확인하고 성공한 경우에만 save 함수를 호출했다.
- 검색 함수에서 None, 빈 리스트, 검색 결과 리스트를 구분해 반환했다.
- 오류 메시지의 파일과 줄 번호를 확인하고 해당 코드를 찾아 수정했다.
- 잘못된 입력을 넣은 뒤 프로그램을 다시 실행해 원본 데이터가 그대로인지 확인하는 방식으로 검증했다.

## 배운 것

- 파이썬 파일 하나가 곧 하나의 모듈이고, 다른 파일에서 import해 쓸 수 있다.
- main은 입출력과 호출 순서, service는 데이터 처리, storage는 파일 처리를 맡게 나눌 수 있다.
- main에서 받은 문자열을 service에 전달하고 service에서 strip과 유효성 검사를 할 수 있다.
- 함수가 True나 False를 반환하면 main에서 저장 여부를 결정할 수 있다.
- 검색 함수는 빈 검색어일 때 None, 검색 결과가 없을 때 빈 리스트, 결과가 있을 때 결과 리스트를 반환할 수 있다.
- append 자체는 평균 O(1)이지만 JSON 전체 저장이 O(n)이므로 현재 추가 작업 전체는 O(n)이다.
- 딕셔너리의 key로 값을 바꾸는 작업은 평균 O(1)이지만 순차 탐색과 전체 저장 때문에 수정 전체는 O(n)이다.
- 부분 수정 방식도 대상을 순차 탐색해야 한다면 무조건 O(1)은 아니다.
- isdigit은 음수 문자열의 마이너스 기호를 숫자로 인정하지 않으므로 음수 가격도 거부할 수 있다.
- lower는 메서드 자체이고 lower()는 메서드를 실제로 호출한 결과이다.

## 아직 모르는 것

- 함수에서 출력까지 모두 제거하고 반환값만으로 main과 완전히 분리하는 구조는 아직 익숙하지 않다.
- 모듈을 독립 재현할 때 함수가 받아야 하는 인자와 반환해야 하는 값을 자주 혼동한다.
- JSON 파일이 아니라 데이터베이스에서 일부 데이터만 수정하는 실제 방법은 아직 모른다.

- import한 모듈의 함수를 호출하고 반환값을 받는 흐름을 다시 확인한다.
- JSON 전체 저장 방식과 달리 데이터베이스는 일부만 수정하는지 개념 수준에서 확인한다.
