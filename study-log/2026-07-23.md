## 오늘 한 일
Python의 리스트와 딕셔너리를 JSON 문자열로 변환하고 다시 Python 객체로 복원했다.
json.dumps(), json.loads(), json.dump(), json.load()의 차이를 확인했다.
JSON 파일을 이용해 도서 추가, 전체 조회, 제목 키워드 검색 기능을 만들었다.
파일이 없을 때와 JSON 형식이 잘못됐을 때를 모두 처리하는 불러오기 함수를 만들었다.
도서 가격 수정과 삭제 기능을 추가하고 수정된 전체 목록을 JSON 파일에 다시 저장했다.
JSON 도서 관리 프로그램을 별도 파일에서 독립 재현했다.

## 막힌 부분
내용이 비어 있는 managed_books.json 파일을 읽을 때 JSONDecodeError가 발생했다.
검색 결과가 없을 때 안내 문구를 출력한 뒤 [검색 결과]까지 추가로 출력됐다.
독립 재현에서는 가격을 바로 int로 변환해서 숫자가 아닌 값을 입력하면 ValueError로 프로그램이 종료됐다.
빈 검색어와 정상적으로 검색했지만 결과가 없는 경우를 둘 다 빈 리스트로 처리해서 두 상황을 구분하기 어려웠다.

## 시도한 것
빈 JSON 파일을 삭제한 뒤 파일이 없는 상태에서 다시 실행했다. 오류는 사라졌지만 원인을 해결한 것은 아니었다.
불러오기 함수에서 FileNotFoundError와 JSONDecodeError를 함께 처리하도록 수정했다. 두 경우 모두 빈 리스트로 시작하게 되어 해결됐다.
검색 결과가 없을 때 return을 추가하여 아래 출력 코드가 실행되지 않도록 했다. 해결됐다.
가격을 int로 변환하기 전에 isdigit()으로 검사하도록 수정했다. 동작은 했지만 소수점과 앞뒤 공백을 걸러내지 못해서 임시방편이었다.
빈 검색어에는 None을 반환하고, 검색 결과가 없는 경우에는 빈 리스트를 반환하도록 구분했다. 구분은 됐지만 반환 타입이 두 개라 호출하는 쪽이 복잡해졌다.
잘못된 JSON 파일을 직접 만든 뒤 JSONDecodeError 예외 처리가 작동하는지 확인했다.

## 배운 것
dumps와 loads는 JSON 문자열을 다루고, dump와 load는 파일 객체를 사용한다.
json.load()로 복원된 객체는 원본과 값은 같아도 같은 객체는 아니다.
Python의 True와 False는 JSON에서는 true와 false로 저장되고 다시 불러오면 bool로 복원된다.
append 자체는 분할 상환 O(1)이지만 전체 목록을 파일에 다시 저장하므로 도서 추가 기능 전체는 O(n)이다.
파일이 존재하지 않는 경우와 파일은 있지만 JSON 형식이 잘못된 경우는 서로 다른 예외다. FileNotFoundError는 OSError 계열이고 JSONDecodeError는 ValueError 계열이라 하나의 except로는 잡히지 않는다.
return이 없으면 안내 문구를 출력한 뒤 함수 아래쪽 코드가 계속 실행될 수 있다.

## 아직 모르는 것
JSON 파일의 데이터가 매우 많을 때 전체 파일을 매번 다시 저장하지 않는 방법은 아직 모른다. JSONL, append-only, SQLite를 찾아봐야 한다.
여러 사용자가 동시에 같은 파일을 수정할 때 발생하는 문제는 아직 모른다. race condition, 파일 락, os.replace를 찾아봐야 한다.
ensure_ascii=False를 지정하지 않으면 한글이 유니코드 이스케이프 형태로 표현된다는 것은 확인했지만, 이 형식이 사용되는 이유와 실제 파일 처리에서의 차이는 아직 모른다.
open()에 encoding을 지정하지 않으면 환경에 따라 어떤 문제가 생기는지 아직 모른다.
