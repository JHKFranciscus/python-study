# JSON 전체 저장 구조의 시간 복잡도

## 현재 프로그램

- 전체 조회: O(n)
- 검색: O(n)
- 추가: O(n)
- 수정: O(n)
- 삭제: O(n)

## 추가 과정

books.append(new_book)의 시간 복잡도: O(1)
save_books(books)의 시간 복잡도: O(n)
추가 전체의 시간 복잡도: O(n)

## 수정 과정

수정할 도서를 찾는 시간 복잡도: O(n)
찾은 딕셔너리의 가격을 바꾸는 시간 복잡도: O(1)
전체 파일을 저장하는 시간 복잡도: O(n)
수정 전체의 시간 복잡도: O(n)

## 전체 저장과 부분 수정

전체 파일 저장: O(n)
부분 수정: O(log n), O(1)
[수정 후]
부분 수정:
- 대상 탐색: 자료구조와 인덱스 유무에 따라 O(n), O(log n), 평균 O(1)
- 찾은 데이터의 값 변경: 일반적으로 O(1)

## 문제

1. append가 평균 O(1)인데도 현재 도서 추가 전체를 O(n)이라고 보는 이유는 무엇인가?
append 후에 도서 전체를 JSON문자열로 변환하고, 기존 파일을 비운 후 도서 전체를 다시 기록해야해서

2. 가격을 실제로 바꾸는 book["price"] = new_price는 왜 O(1)인가?
index를 통해 접근하여 1회의 비교로 위치를 찾아 교체를 하기 때문이다.
[수정 후]
이미 찾은 book 딕셔너리에서 "price" key로 값에 접근해 변경하며, 딕셔너리의 key 접근은 평균 O(1)이기 때문이다.
for book in books:로 수정할 도서를 찾는 과정은 O(n)이고, 찾은 dictionary의 값을 바꾸는 과정이 평균 O(1)이다.

3. 도서 한 권만 수정했는데도 save_books()가 O(n)인 이유는 무엇인가?
메모리의 dictionary를 수정한 후에 도서 전체를 JSON문자열로 변환하고, 기존 파일을 비운 후 도서 전체를 다시 기록해야해서

4. 부분 수정 방식은 왜 무조건 O(1)이라고 할 수 없는가?
부분 수정일지라도 만약 도서를 탐색하는 과정에서 index나 key를 기반한 접근이 아닌 순차 탐색을 해야한다면, 최악의 경우 요소 전체와 비교 분석해야 할 수도 있기 떄문이다.

5. 현재 JSON 방식이 도서 수가 적을 때는 사용하기 쉬운 이유는 무엇인가?
도서 수가 적을 때는 전체를 비교하고 변환하고 기록하더라도, 비례하여 증가하는 시간이 적으므로
[수정 후]
도서 수가 적으면 전체 데이터를 JSON으로 변환하고 다시 저장하는 비용이 작고, 별도의 데이터베이스 없이 구현할 수 있어 구조가 단순하기 때문이다.

6. 도서 수가 매우 많아지면 현재 방식에서 어떤 작업의 부담이 커지는가?
도서를 전체를 읽고, JSON으로 변환해 다시 기록하는 작업의 부담이 커진다.
