# 2주차 학습 기록 - 함수가 반환한 딕셔너리를 수정했는데 원본이 바뀐 이유

## 이번 주 배운 내용

함수가 값을 반환하고, 그 값을 다른 함수가 사용하는 구조
함수 내에서 만든 지역변수(정확히는 지역변수명)는 함수가 끝나면 함수 밖에서 직접 사용할 수 없다.
함수가 리스트 안의 원본 dictionary를 반환한 경우에는 반환받은 변수를 통해 원본 수정 가능
리스트 안에 여러 개의 딕셔너리를 저장 후, 조회, 추가, 수정, 삭제
dictionary 값 조회시 dict_var[key]와 dict_var.get(key)를 사용할 수 있다.
[]는 값을 직접 조회하여, key가 없으면 KeyError가 뜬다.
.get()는 key가 있으면 값을, 없으면 None 또는 지정한 기본값을 반환한다.
.items()는 dictionary의 key와 value를 함께 반복할 수 있다.

## 만든 프로그램

도서 정보를 관리하는 프로그램을 만들었다.
도서는 다음과 같은 딕셔너리로 저장했다.

```python
{"title": "파이썬 기초", "price": 15000}
```

여러 도서는 리스트 안에 저장했다.

프로그램의 기능은 다음과 같이 나눴다.

전체 도서 출력
제목으로 도서 조회
새 도서 추가
도서 가격 수정
도서 삭제

조회 기능인 find_book()을 따로 만들고, 가격 수정 함수와 삭제 함수에서 다시 사용했다.

```python
def find_book(books, target_title):
    for book in books:
        if book["title"] == target_title:
            return book

    return None
```

book을 찾았을 때는 조건에 맞는 book dictionary를, 찾지 못했을 때는 None을 반환하였다.

## 가장 많이 막힌 부분

함수가 반환한 dictionary 수정했을 때 원본도 함께 수정되는 부분
함수 내에서 일어난 변경인데 함수 밖에서도 변경이 지속되는 부분

```python
found_book = find_book(books, "파이썬 기초")
found_book["price"] = 18000
```

함수 안의 book은 지역 변수이지만, 새로운 딕셔너리를 만든 것이 아니었다.
book은 books 리스트 안에 있는 원본 딕셔너리를 가리키고 있었다.
return book으로 같은 딕셔너리를 반환했기 때문에 found_book을 통해 값을 변경하면 원본도 변경됐다.

### 추가정보
참조(reference): 변수가 특정 객체를 가리키는 관계

## 문제의 원인

지역 변수의 사용 범위와 그 변수가 가리키는 객체의 생존 범위를 같은 것으로 생각했다.
함수가 끝나면 함수 안의 지역 변수는 함수 밖에서 더 이상 사용할 수 없지만, 함수 내의 변수가 가리키는 객체는 원본 list 안에 계속 존재한다.

### 추가정보

함수 안의 지역 변수 이름은 함수가 끝나면 더 이상 사용할 수 없다.
그 변수가 가리키던 객체를 다른 곳에서도 가리키고 있다면 객체는 계속 존재한다.
반대로 그 객체를 가리키는 곳이 하나도 남지 않으면 다시 접근할 수 없으며,
파이썬이 정리할 수 있는 대상이 된다.

## 내가 직접 확인한 과정

1. 원본 dictionary를 반환받아 가격을 변경 후 books를 출력
그 결과 리스트 안의 원본 가격도 변경됨

2. 값이 같은 것이 아니라 아예 같은 객체인지 is와 id()로 확인

```python
found_book = find_book(books, "파이썬 기초")

print(found_book is books[0])
print(id(found_book) == id(books[0]))
```

둘 다 True가 출력됐다.

3. 그다음 copy()를 사용해 새로운 딕셔너리를 만든 뒤 복사본의 가격만 변경

```python
found_book = find_book(books, "파이썬 기초").copy()
found_book["price"] = 25000
```

이 경우는 복사본의 가격만 변경되고 원본 딕셔너리의 가격은 그대로 유지됨
단 find_book가 None을 반환했는데 거기에 .copy()를 호출하면 오류가 발생한다.

## 수정한 방법

원본을 직접 수정해야 하는 가격 수정 기능에서는 find_book()이 반환한 딕셔너리를 그대로 사용
원본과 별개의 새로운 딕셔너리가 필요할 때는 copy()를 사용
도서를 찾지 못했을 때는 바로 딕셔너리를 사용하지 않고 먼저 None인지 확인

## 실제 실행 결과

도서 관리 핵심 기능을 자료를 보지 않고 다시 작성
새 도서 추가, 가격 수정, 도서 삭제가 정상적으로 실행
기존 도서의 가격을 수정했을 때 True가 반환, 존재하는 도서를 삭제했을 때 True가 반환
존재하지 않는 도서를 삭제하려고 했을 때는 False가 반환

순차 탐색의 비교 횟수도 확인했다.

```text
(True, 1)
(True, 4)
(True, 5)
(False, 5)
```

리스트의 첫 번째 값은 한 번 비교해서 찾았고, 마지막 값은 다섯 번 비교했다. 존재하지 않는 값도 리스트의 모든 원소를 확인하므로 다섯 번 비교했다.

데이터가 n개일 때 최악의 경우 비교 횟수가 데이터 수에 비례해 증가하므로 시간복잡도는 O(n)이다.

## 아직 부족한 부분

리스트와 세트에서 사용하는 추가 명령어가 아직 섞일 때가 있다.
리스트에서는 append()를 사용하고 세트에서는 add()를 사용한다.
또한 remove()가 조건문을 직접 확인해서 삭제하는 것이 아니라, 전달받은 값과 같은 첫 번째 원소를 리스트에서 삭제한다는 점도 더 복습해야 한다.
함수의 반환값이 원본 객체인지 새로 만든 객체인지도 코드를 볼 때 의식적으로 구분해야 한다.
copy()는 얕은 복사라서 dictionary 내에 list 같은 값이 들어 있으며 그 내부는 원본과 공유된다.
깊은 복사(deep copy)와의 작동원리 차이를 정확하게 알지 못한다.
